# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository layout

This repo contains the backend (`backend/`) and the frontend (`frontend/`), built separately and in parallel by another engineer against mock data, per `frontend/API_CONTRACT.md`. All backend commands below assume `cd backend` first. The frontend is now wired to this real backend via an additive adapter layer — see "Frontend integration adapter layer" below — rather than any change to the frontend's own code.

## Commands

```bash
# Setup (one-time)
python3 -m venv .venv && ./.venv/bin/pip install -r requirements.txt
cp .env.example .env

# Local infra (Postgres on host port 5433, Redis on 6379 — see docker-compose.yml)
docker compose up -d postgres redis

# Migrations
./.venv/bin/alembic upgrade head
./.venv/bin/alembic revision --autogenerate -m "description"

# Run the API
./.venv/bin/uvicorn app.main:app --reload
# -> docs at http://127.0.0.1:8000/docs, liveness at /health, readiness at /api/v1/health/ready

# Tests (SQLite fixtures — no Docker/Postgres/Redis needed to run the suite)
./.venv/bin/pytest -q
./.venv/bin/pytest tests/test_auth.py -q                              # single file
./.venv/bin/pytest tests/test_auth.py::test_register_login_and_me -q  # single test
./.venv/bin/pytest tests/continuum -q                                 # single package
```

Generating a migration requires local Postgres running (`docker compose up -d postgres`) — `alembic revision --autogenerate` connects to it to diff against the current models.

There is no configured linter/formatter yet.

## Architecture

FastAPI + async SQLAlchemy 2 + PostgreSQL + Redis, structured as strict layers:

```
router (app/api/v1/endpoints/) -> service (app/services/) -> repository (app/repositories/) -> model (app/models/)
```

Routers only validate input, call a service, and return a response — they never contain business logic or touch the ORM directly. Services hold business logic and call repositories for data access. Repositories subclass the generic `BaseRepository[ModelType]` (`app/repositories/base.py`) and add model-specific queries.

**Response envelope.** Every endpoint returns one of two shapes via helpers in `app/core/responses.py`: `success_envelope(data, message)` -> `{success: true, data, message, timestamp}`, or the error shape `{success: false, error: {code, message}}` produced automatically by the exception handlers in `app/core/exceptions.py` whenever an `AppException` subclass (`NotFoundError`, `ConflictError`, `UnauthorizedError`, `ForbiddenError`, `ValidationAppError`) is raised. Raise these instead of returning error responses manually; never let a raw exception/stack trace reach the client — `app/core/exceptions.py` has a catch-all handler for that.

**Auth.** JWT access + refresh tokens (`app/security/jwt.py`), bcrypt password hashing called directly (`app/security/password.py` — deliberately not `passlib`, which is incompatible with bcrypt >= 4.2), and `get_current_user` (`app/security/dependencies.py`) as the `Depends()` for protected routes.

**Database sessions.** `app/database/deps.py` provides `get_db` — an async-session-per-request dependency that commits on success and rolls back on exception. All models use the `UUIDPrimaryKeyMixin` / `TimestampMixin` from `app/database/base.py`.

**Config.** Pydantic `Settings` (`app/config/settings.py`) loaded from `.env` via `get_settings()` (lru-cached). Model/provider config for Continuum is **not** here — see below.

### Continuum — the AI orchestration layer

**This is a Continuum AI hackathon project.** Continuum is the real `shyftlabs-continuum` SDK (`git@github.com:shyftlabs/continuum.git`, installed via `requirements.txt`) — not something written in this repo. The one hard architectural rule: **no router, service, or agent may import `anthropic`/`openai`/`google-genai` (or any model SDK) directly.** Every AI capability flows through:

```
router -> service -> ContinuumClient -> Agent -> Continuum SDK (AgentRunner) -> configured model -> typed response
```

- `app/continuum/base_agent.py` — `BaseAgent[InputT, OutputT]`, the generic ABC every domain agent subclasses. Constructor takes `continuum_client: ContinuumClient` (not a provider/model client of any kind).
- `app/continuum/registry.py` — `AgentRegistry` (name -> agent class) + `@register_agent("name")` decorator. Unchanged since this was first built.
- `app/continuum/client.py` — `ContinuumClient`, the **only file in this app that imports `continuum.agent`/`continuum.core`**:
  - `run_agent(name, input_data)` / `run_workflow(names, input_data)` / `health_check()` / `register_agent()` — registry-dispatch API that services call. Returns `AgentExecutionResult` (`success`, `data`, `error`, `error_type`, `duration_ms`). No retry loop here — the SDK's own `AgentRunner`/`RunnerConfig` already retries and circuit-breaks internally, so don't add a second layer of retry on top.
  - `run_prompt(agent_name, instructions, input_text, output_schema=None, model=None)` — what domain agents call to actually reach a model. Builds a `continuum.agent.BaseAgent` + runs it via a module-level `AgentRunner`, with `ContainerConfig(enable_memory=False, enable_session=False, enable_langfuse=False)` — Sentinel's calls are single-shot/stateless, so no Redis/Qdrant/Langfuse is needed. Maps the SDK's exception hierarchy and `AgentResponse.structured_output_error` into `app/continuum/exceptions.py`'s own `ContinuumTimeoutError` / `ContinuumUnavailableError` / `MalformedResponseError`.
  - **Gotcha:** never pass `model=None` to the SDK's `BaseAgent(...)` — its dataclass default (`settings.default_llm_model`) only applies when the field is *omitted*, not when explicitly `None`. `run_prompt` builds its kwargs dict conditionally for exactly this reason.
- `app/continuum/dependencies.py` — `Depends(get_continuum_client)`.

**Model/provider configuration lives entirely in the Continuum SDK's own env vars** (not `CONTINUUM_*`, not `app/config/settings.py` — see `.env.example`): `DEFAULT_LLM_MODEL`, and this hackathon's shared **Smart Gateway** (`SMART_GATEWAY_URL=https://continuum.shyftops.io/v1` + `SMART_GATEWAY_API_KEY`), which routes every model call through one endpoint so no per-provider key is needed. The SDK calls `load_dotenv()` itself on import, so anything in `backend/.env` reaches it automatically — verified empirically, not assumed.

**Gotcha confirmed by reading the installed SDK source, not assumed:** once `SMART_GATEWAY_URL` is set, `DEFAULT_LLM_MODEL`'s literal value is discarded — `GatewayProvider._normalize_model()` replaces any model string that isn't already `auto/<tier>` with one derived purely from `SMART_GATEWAY_DEFAULT_MODE` (`strict`→`cheap`, `modest`→`mid` — the SDK default, `quality`→`quality`). The gateway server then picks the actual concrete model within that tier itself; there is no client-side way to pin a specific model (e.g. a particular Claude/GPT/Gemini version) while going through the gateway. Set `SMART_GATEWAY_DEFAULT_MODE=quality` in `.env` to bias toward stronger models — currently set for this project — but the exact model served per call is still the gateway's call, not ours. **Also confirmed by reading the log line the "🔀 Smart Gateway routing" log statement prints is cosmetic** — it echoes the pre-normalization `DEFAULT_LLM_MODEL` string whenever that string happens to contain a `/`, regardless of what's actually sent over the wire. The only trustworthy signal for what actually served a call is the later `🎯 Gateway selected model: X` line, logged from the real HTTP response — and it never appears at all if the call times out first.

**Second gotcha, same investigation:** the SDK's own defaults are `llm_request_timeout=60` (seconds, per attempt) and `llm_max_retries=3` — three 60s attempts plus small backoff delays add up to the ~187-188s total that repeated `LLM_TIMEOUT` failures showed in testing, every time, suspiciously consistently. A `quality`-tier model that genuinely needs more than 60s to produce a full structured PR review will fail all three attempts identically. Widened via `LLM_REQUEST_TIMEOUT=120` + `LLM_MAX_RETRIES=2` in `.env` (bounds the worst case to ~240s instead of ~180s-but-more-likely-to-actually-succeed-once).

**Agents live under `app/agents/<domain>/`**, grouped by product module (`development/`, `production/`, `executive/`). The reference pattern is `app/agents/development/pr_review_agent.py` + its sibling `prompts.py` (dedicated prompt module — never inline large prompt strings in the agent) and `schemas.py` (typed I/O, no `dict`/`Any` in outputs). To add a new agent: define Input/Output Pydantic models, write its prompt in its own `prompts.py`, subclass `BaseAgent`, call `self.continuum_client.run_prompt(..., output_schema=YourOutput)` in `run()`, decorate with `@register_agent("your_agent_name")`, and add the import to `app/agents/__init__.py`. Callable via `continuum_client.run_agent("your_agent_name", input_data)` with no other wiring.

**Direct-provider fallback (`app/continuum/fallback.py`), added for live-demo reliability.** Given the ~187-188s Continuum failure mode above, `ContinuumClient.run_prompt()` now races the Continuum call against `CONTINUUM_FALLBACK_TIMEOUT_SECONDS` (default 30s, Sentinel's own setting in `app/config/settings.py` — not a Continuum SDK var) and, on *any* Continuum-side failure (timeout, malformed output, agent error — not just slowness), falls through to a direct OpenAI call, then a direct Anthropic call, each bounded by its own 20s timeout. This is the one deliberate, explicit exception to "no router/service/agent imports a model SDK directly" — and even here, the import is confined to `app/continuum/fallback.py`, so `ContinuumClient` (via this module) remains the *only* thing in the app that ever touches a model SDK; callers (`PRReviewService`, agents) still never know or care which path served a call. Purely additive: with no `OPENAI_API_KEY`/`ANTHROPIC_API_KEY` set, `run_with_fallback()` re-raises the original Continuum exception unchanged, so behavior for anyone without those keys is identical to before this existed. **Currently both keys are unset in `.env`** (OpenAI hit `insufficient_quota`, Anthropic was never funded) — the fallback code stays in place since it's harmless when idle, but in practice right now Continuum is the only path that actually serves a call.

**Third gotcha — standalone-diagnosed, not guessed.** Built a one-off script (`AgentRunner` + `BaseAgent` called directly, bypassing our app entirely) to answer "does Continuum even work?": 6 queries, easy → a real PR diff with structured output. Findings, in order of how much they mattered:
1. Continuum works fine and does real dynamic model routing within a tier — confirmed live: trivial arithmetic → `gemini-3-flash-preview`, a haiku/CAP-theorem question → `claude-sonnet-4-6`, a free-text code security review → `claude-opus-4-7` (correctly caught a SQL injection). All four plain-text calls succeeded in under 15s.
2. A **real schema bug**, not a model failure: asking for structured `PRReviewOutput` on a trivial one-line diff, the model produced a fully correct review — then failed validation because it labeled a stylistic note `severity: "info"`, a value `FindingSeverity` didn't allow (only low/medium/high/critical). One bad enum value voided an otherwise-perfect response after a retry. Fixed by adding `INFO` to `FindingSeverity` in `app/agents/development/schemas.py` — confirmed via grep that nothing downstream assumes exactly 4 severity values.
3. **The real, un-fixable-by-us cause of the long timeouts**: replaying the *exact* diff from a real PR that had timed out live, through the exact same `PRReviewOutput` schema, returned an actual `504 Gateway Time-out` — an HTML error page from the Smart Gateway's own reverse proxy, not our client-side `LLM_REQUEST_TIMEOUT`. That's a hard ~180s ceiling on the shared hackathon gateway infrastructure itself; raising our own client-side timeout past it does nothing, since the server hangs up first. `quality` tier apparently routes structured PR reviews to slower models (Opus-class) that can need more than that ceiling to finish. Switched `SMART_GATEWAY_DEFAULT_MODE` to `modest` — faster models, reliably under the ceiling, less review depth. Reliability over max quality for a live demo.

### AI-powered PR review

`POST /api/v1/github/repositories/{repository_id}/pulls/{pr_number}/review` — live-verified end-to-end against a real repo and the real Smart Gateway (not just mocked tests). Flow: `app/services/pr_review_service.py` re-fetches the PR's latest metadata/diff/files/commits via the existing `GitHubIntegrationService` (no GitHub logic duplicated), runs `pr_review` through `ContinuumClient`, persists a summary to `ai_reviews` (`app/models/review.py` — never the diff, only `repository_id`/`pull_request_number`/`review_timestamp`/`summary`/`risk_score`/`deployment_confidence`/`recommendation`/`technical_debt_summary`), and returns the full structured `PRReviewOutput` (`app/agents/development/schemas.py`). Continuum failures map to specific HTTP codes: malformed structured output -> 422, timeout -> 504, unavailable/misconfigured -> 503.

**Known, deliberately-accepted gap (not fixed, decided against fixing for now):** this endpoint, and its frontend-adapter twin `POST /api/development-intelligence/repositories/{id}/pulls/{n}/review`, are both still **synchronous** and both still request the full 12-section `PRReviewOutput` schema — the exact combination proven (see the Continuum gotchas above) to risk a real ~180s gateway timeout that no client-side setting can prevent. The frontend no longer calls either of them (the "Run review" button now redirects to the async, lean-schema `/compare` flow instead — see below), so neither is on the UI's critical path today, but both remain real, callable, and exposed the same way they always were. A side effect worth knowing: since nothing in the current UI flow triggers a full review anymore, `TechnicalDebtService` and the AI-fixes list (both derived from a full `PRReviewOutput`) stop getting *new* data — existing persisted rows still show, but nothing new is generated unless something calls one of these two endpoints directly. If the deployment-safety fix below is ever wanted here too, the pattern is already proven out in `/compare`'s implementation.

### GitHub integration

`app/integrations/github/client.py` (`GitHubClient`) is a typed async wrapper over the GitHub REST API — it is a plain data-fetching integration, deliberately **not** routed through Continuum (no AI capability involved). It takes a `transport: httpx.BaseTransport | None` constructor param specifically so tests can inject `httpx.MockTransport` instead of hitting the network.

Two new tables (`app/models/integration.py`, `app/models/repository.py`) support it: `Integration` holds one connected GitHub PAT per row, encrypted via `app/security/encryption.py` (Fernet; the key is derived from `JWT_SECRET_KEY` via SHA-256 — there is no separate encryption secret to configure). `TrackedRepository` holds repos a user has explicitly opted into tracking. **Pull requests, diffs, files, and commits are always fetched live from GitHub and never persisted** — only the integration connection and the set of tracked repos are stored. `app/services/github_service.py` is the only place that translates `GitHubError` subclasses into the app's standard `AppException` subclasses (`GitHubAuthError` -> `UnauthorizedError`, `GitHubNotFoundError` -> `NotFoundError`, `GitHubRateLimitError` -> `ForbiddenError`, anything else -> `ConflictError`).

### Docker monitoring

`app/integrations/docker/` — targets the **local** Docker Engine only (Sentinel and its dependencies all run as containers on the same single EC2 host; no remote Docker/Kubernetes/ECS support, though the schemas are shaped to allow that later without redesign). Pure data collection, deliberately **not** routed through Continuum, and **nothing is persisted** — every call hits the Docker daemon live, on purpose, so a future Root Cause Analysis Agent always sees current state.

- `client.py` — `DockerClient` wraps the official `docker` SDK (`docker.from_env()`, never the CLI) via `asyncio.to_thread` (the SDK is synchronous). Takes an injectable `docker_client` param, same test-seam pattern as `GitHubClient`'s `transport` param — unit tests never need a real daemon.
- `service.py` — `DockerMonitoringService` does all the interpretation: Docker's own CPU%/memory-usage formulas from a raw `stats()` dict, RFC3339Nano timestamp parsing (including Docker's `0001-01-01` zero-value meaning "never happened"), and maps `DockerError` subclasses to the app's standard `AppException`s.
- `schemas.py` — `ContainerSummary` (list-weight), `ContainerDetail` (summary + mounts + an embedded stats snapshot), `ContainerStats`, `ContainerLogs` (parsed into timestamped lines, not a raw blob). Deliberately shaped for a future AI agent to consume directly.
- API under `/api/v1/docker/`: `GET /containers`, `GET /containers/{id}` (metadata + one stats snapshot), `GET /containers/{id}/stats` (just stats, for fast polling), `GET /containers/{id}/logs`, `POST /containers/{id}/restart` (safe restart only — no kill/remove/prune).

### Root Cause Analysis

`POST /api/v1/docker/containers/{id}/analyze` (`app/services/root_cause_service.py`) — same reference pattern as PR review, one layer deeper: the `root_cause` agent (`app/agents/production/root_cause_agent.py`, `prompts.py`, `schemas.py`) **never calls Docker itself** — `RootCauseAnalysisInput` embeds `ContainerDetail` and `ContainerLogs` straight from `app/integrations/docker/schemas.py`, so all Docker access stays inside `DockerMonitoringService` (`get_container()` for metadata+stats, `get_container_logs()` for logs) with zero duplicated Docker logic. Diagnosis only — this phase never restarts a container; `RecoveryPlan`'s `auto_restart_safe`/`requires_human_intervention` fields are shaped for a future Safe Recovery phase to act on, not for this one to. Logs `docker_collection_ms` separately from `continuum_latency_ms` so slow-Docker vs. slow-model is distinguishable. Error mapping duplicates `pr_review_service.py`'s small `_map_agent_failure` static method rather than extracting a shared helper — deliberate, since PR review is a "complete, stable" module per Phase 5's instructions and wasn't touched.

**Update:** early testing saw the model call time out twice (~185s) against a container with very repetitive logs. A later run against a different, healthy container succeeded with a well-reasoned diagnosis (correctly found no failure, 75% confidence, correctly recommended against restarting a healthy container) — both the pipeline and the model's judgment are now confirmed live.

### Incident Engine

The canonical production object (`app/models/incident.py`, `app/schemas/incident.py`, `app/repositories/incident_repository.py`, `app/services/incident_service.py`, `app/api/v1/endpoints/incidents.py`). Sentinel thinks in incidents, not "analyze this container" — but the incident engine never duplicates Docker access, AI reasoning, or restart logic; it composes `DockerMonitoringService` and `RootCauseAnalysisService` as black boxes.

- **Lifecycle** (stored as a plain string column, not a DB enum, matching the rest of the codebase's convention): `open -> investigating -> analyzed | recovery_available -> resolved`. `analyzed` vs `recovery_available` is decided by the RCA agent's own `recovery_plan.auto_restart_safe` — the incident service doesn't second-guess it. `POST /{id}/recover` raises `ConflictError` (409) unless status is exactly `recovery_available`.
- Two child tables via SQLAlchemy relationships (`cascade="all, delete-orphan"`): `IncidentContainer` — a **point-in-time snapshot** (container_id/name/image), not a live FK to anything Docker-side, since Docker state can change or the container can disappear entirely. `IncidentEvent` — the append-only timeline.
- **Gotcha reused from earlier phases**: when appending new `IncidentContainer`/`IncidentEvent` rows to `incident.affected_containers`/`incident.events`, never pass `incident_id` explicitly — the relationship sets the FK automatically at flush time. Passing `incident.id` manually before the parent has been flushed would silently write `None` (SQLAlchemy's `default=uuid.uuid4` on the PK is a client-side default evaluated at flush, not at object construction).
- `analyze_incident()` always targets `incident.affected_containers[0]` (the "primary" container) — multi-container aggregation is a deliberate non-goal for this phase, not an oversight.
- `recover_incident()` calls `DockerMonitoringService.restart_container()` directly — the exact same method backing `POST /docker/containers/{id}/restart`, not a reimplementation.
- Live-verified end to end against real Postgres + the real local Docker Engine + a real Continuum call, including confirming the `recovery_available`-only guard on `/recover` holds under real (not mocked) conditions.

### Slack incident notifications

`app/integrations/slack/` — posts a formatted message to a Slack channel whenever `IncidentService.analyze_incident()` completes, so a collapsing container surfaces somewhere a human is actually watching instead of only in the dashboard. Pure data delivery, deliberately **not** routed through Continuum (no AI capability involved) — the diagnosis it's formatting was already produced by the `incident_diagnosis` agent (see the health monitor section below) earlier in the same call.

- **Incoming Webhook, not a bot.** `SlackClient` (`app/integrations/slack/client.py`) POSTs `{text, blocks}` to a single webhook URL (`SLACK_WEBHOOK_URL`, Sentinel's own setting in `app/config/settings.py` — not a Continuum var). Chosen over a full Slack bot/OAuth app specifically to avoid standing up scopes, a signing secret, and a public HTTPS endpoint for a hackathon MVP. Same injectable-transport test seam as `GitHubClient`/`DockerClient`.
- **3-level severity, not the diagnosis agent's 4.** `IncidentDiagnosis.severity` is `low/medium/high/critical` (`app/agents/production/schemas.py`); the Slack message collapses this to `1/2/3` (`_SEVERITY_LEVEL` in `app/services/incident_service.py` — high and critical both read as `3`, with the original label still spelled out in the text alongside the number) since a 3-level scale was the explicit ask.
- **No interactive "apply" button.** A real clickable Slack action would require a Slack app with an Interactivity Request URL (a public endpoint plus request-signature verification) — out of scope for now. Instead the message text itself says whether recovery is auto-safe and, if so, exactly which call applies it (`POST /api/v1/incidents/{id}/recover`) — the same endpoint the dashboard's own "Recover" button calls, no logic duplicated.
- **Best-effort, never fails the analysis.** `IncidentService._notify_slack()` is called after `analyze_incident()` has already updated and flushed the incident; a missing webhook (`SlackClient.is_configured` is `False`) is skipped silently, and any exception during the actual POST (webhook down, bad URL) is caught and logged as a warning — same "degrade quietly, never crash on an optional external dependency" convention as the AWS cost integration. Analysis always succeeds or fails on its own merits, independent of Slack.
- **Automatic detection is now wired** via the health monitor below — see that section for how a collapse turns into an incident with zero human action.

### Health monitor (automatic incident detection)

`app/services/health_monitor.py` (`HealthMonitor`) — a single background `asyncio` task, started/stopped from `app/main.py`'s `lifespan`, that closes the loop the Incident Engine originally left open: something still had to notice a container had collapsed and call `POST /incidents` manually. Now nothing does — the monitor polls `DockerMonitoringService.list_containers()` every `HEALTH_MONITOR_INTERVAL_SECONDS` (default 20s) and, for anything it considers collapsed (`health == unhealthy`, or not `running` and `status` in `exited`/`dead`), drives it through the exact same `IncidentService.create_incident()` → `analyze_incident()` calls the manual API path uses — no detection-specific business logic, Docker access, AI reasoning, or Slack formatting is duplicated here.

- **A second, live-confirmed instance of the schema-size timeout gotcha, this time on `root_cause`.** Live-testing this exact flow (stop `backend-redis-1`, let the monitor detect it) hit a real `504 Gateway Time-out` from the Smart Gateway on `IncidentService.analyze_incident()`'s call into `RootCauseAnalysisService.analyze_container()` — the same nested-schema-with-list-fields shape (`RootCauseAnalysisResponse`: 3 nested objects, `evidence`/`recommended_actions` as object lists) that caused the original PR-review timeouts, now confirmed on a second, unrelated agent. Repeated once, failed identically both times — not a fluke.
- **Fixed with the same proven pattern as `pr_compare_review`.** `IncidentDiagnosis` (`app/agents/production/schemas.py`) + the `incident_diagnosis` agent (`app/agents/production/incident_diagnosis_agent.py`) are a deliberately flatter alternative — 8 flat fields, `recommended_actions` as a `list[str]` instead of `list[Recommendation]` objects, no nested `incident_summary`/`root_cause`/`recovery_plan` wrapper objects. `RootCauseAnalysisService` gained a parallel `diagnose_incident()` method (alongside the original, untouched `analyze_container()`) that calls this leaner agent; `IncidentService.analyze_incident()` was switched to call `diagnose_incident()`. The original `root_cause` agent/`RootCauseAnalysisResponse`/`analyze_container()` are completely unchanged and still back the on-demand `POST /docker/containers/{id}/analyze` endpoint.
- **Live-verified the fix, not just asserted it**: replayed the identical failing case (same container, same logs) immediately after the change — the lean schema call succeeded in ~14 seconds (routed to `claude-opus-4-7` by the gateway, not a weaker model — reliability came from the schema shape, not a quality tradeoff) and produced a correct, well-reasoned diagnosis (correctly recognized a clean `SIGTERM` shutdown rather than a crash, and correctly judged auto-restart safe).

- **Dedup via the database, not memory.** `IncidentRepository.find_active_incident_for_container()` (a new query, joins `IncidentContainer` and filters `Incident.status != resolved`) is checked before opening a new incident, so a container stuck in a crash loop for 10 poll cycles produces exactly one incident, not ten — and since the check is DB-backed rather than an in-memory set, it survives an app restart too.
- **Fresh resources per tick, same reasoning as the compare background job.** Each poll builds its own `AsyncSessionLocal()` session, `ContinuumClient`, and `SlackClient` — there's no request-scoped `Depends()` to reuse here any more than there is in a `BackgroundTask`.
- **One bad tick, one bad container, never kills the loop.** Both the outer poll loop and the per-container analysis are individually wrapped in `try/except Exception` — a Continuum failure analyzing one collapsed container logs a warning and moves on to the next container and the next tick, rather than silently ending all future detection.
- `HEALTH_MONITOR_ENABLED=false` disables the whole thing (e.g. for a shared Docker host running unrelated containers where auto-opening incidents for someone else's container would be wrong). Not started at all under the test suite's `ASGITransport` (which never triggers FastAPI's `lifespan`), so pytest runs never touch a real Docker daemon because of this.

### Engineering health score & AWS cost

`app/services/executive_service.py` (`ExecutiveMetricsService`) computes a real, first-pass-and-documented-as-such engineering health score: 40% incident resolution rate + 35% container health % + 25% average PR deployment confidence, each from real repository/service data, defaulting to 100 when there's no data yet. `app/integrations/aws/` (`AWSCostClient`) wraps boto3's Cost Explorer for real monthly cost-by-service — this dev environment has no valid AWS credentials, so every consumer treats `AWSCredentialsError`/`AWSUnavailableError` as "no cost data yet" and degrades to zero rather than crashing or faking numbers.

### Technical debt & AI fixes

`app/models/ai_fix.py` persists `AIFix` rows as a side effect of `PRReviewService.review_pull_request()` reading its own `code_fix_suggestions` — no separate AI call. `app/services/technical_debt_service.py` derives technical debt items from already-persisted `AIReview.technical_debt_summary` text rather than running a second Continuum agent, since every PR review already produces real AI-written debt commentary.

### Frontend integration adapter layer

The real frontend (`frontend/`) expects flat, camelCase, unenveloped JSON exactly matching `frontend/API_CONTRACT.md` — the opposite of `/api/v1`'s enveloped snake_case convention. Rather than touch any stable `/api/v1` code, this is a wholly separate, additive package mounted alongside it:

```
router (app/api/frontend/*.py) -> existing /api/v1 services & repositories, reused as black boxes -> CamelModel response
```

- `app/schemas/camel.py` — `CamelModel` (`alias_generator=to_camel`). **Gotcha:** `to_camel` doesn't preserve multi-letter acronyms (`open_prs` -> `openPrs`, not `openPRs`) — every contract field with an embedded acronym like "PR" needs an explicit `Field(serialization_alias="openPRs")` override.
- `app/utils/time.py` — the contract's timestamp fields are **display strings**, not ISO dates: `format_relative_time()` ("12m ago"), `format_clock_time()` ("10:42:01"), `format_duration_since()` ("14d 6h"), `format_plain_date()` ("2026-06-18").
- One file per contract section under `app/api/frontend/` — `auth.py`, `integrations.py`, `settings.py`, `development.py`, `production.py`, `executive.py`, `dashboard.py` — aggregated by `app/api/frontend/router.py` and mounted at `/api` in `app/main.py` (alongside `/api/v1`, not instead of it). Each reuses the same services/repositories as the corresponding `/api/v1` feature: development reuses `PRReviewService`/`GitHubIntegrationService`, production reuses `DockerMonitoringService`/`IncidentRepository`, executive reuses `ExecutiveMetricsService`/`AWSCostClient`, and the dashboard composes all three's underlying data sources directly (not their HTTP endpoints) for one aggregated landing-page response.
- Documented, deliberate deviations from a literal contract reading rather than silent shortcuts: the development summary never auto-triggers a real review for any PR — every PR shows `reviewed: false` with a placeholder risk/confidence; the dashboard never triggers a fresh review at all (cached reviews only, since it has to stay fast); `healthTrend` in both executive and dashboard responses is a single current-day point, not a real multi-month/7-day series, since no historical health-score snapshots are persisted yet.
- **Branch compare (`/development-intelligence/compare`) is the one place a real AI review runs on every call, and it's the whole reason for two pieces of infrastructure built specifically for it:**
  - **Lean schema, dedicated agent.** `CompareReviewOutput` (`app/agents/development/schemas.py`) + the `pr_compare_review` agent (`app/agents/development/compare_review_agent.py`) are a deliberately smaller alternative to the full `PRReviewOutput`/`pr_review` — 6 flat fields and one merged findings list instead of 12 sections and 4 separate finding lists. Empirically proven necessary: the full schema reliably hit the Smart Gateway's ~180s reverse-proxy ceiling on real diffs regardless of tier; this lean one reliably finishes in ~30-40s. `PRReviewOutput`/`pr_review` itself is untouched — nothing about the stable PR review feature changed.
  - **Async job + polling, not a held-open request.** `GET /compare` was replaced with `POST /compare/jobs` (returns a `jobId` immediately, 202) and `GET /compare/jobs/{jobId}` (poll for status). The real work runs in a FastAPI `BackgroundTask` (`app/core/jobs.py` is the generic Redis-backed job store) with its own freshly-constructed DB session and Continuum client — a background task runs after the response is already sent, so the original request's `Depends()`-injected `db`/services are already torn down and can't be reused. This matters even beyond the Smart Gateway's own ceiling: most production load balancers/reverse proxies kill idle connections around 30-100s by default, which no `LLM_REQUEST_TIMEOUT`-style app setting can influence. On the frontend, this complexity is fully encapsulated inside `compareBranches()` (starts the job, polls every 2s up to ~260s) — `useBranchComparison`/`BranchComparisonPanel` needed zero changes, since the function's external contract (`Promise<BranchComparison>`) didn't change.
  - The "Run review" button in the PR list no longer calls a dedicated review endpoint at all — it switches to the Branch Comparison tab with that PR's real `base`/`head` branches (from `GitHubPullRequestSummary.base_branch`/`head_branch`, already present in the list-PRs API response, zero extra GitHub calls) pre-filled, and the result flows back into the PR list's cache once it resolves. Rationale: a 30s wait behind a tiny inline button reads as broken; the same wait on a dedicated comparison view reads as expected.
- **Continuum response cache (`app/continuum/cache.py`)**, wired into `ContinuumClient.run_prompt()` itself so every agent benefits, not just compare: a Redis-backed cache keyed by a hash of `(agent_name, instructions, input_text, output_schema)` — an identical request returns the prior result instantly instead of paying for another AI call. 24h TTL (`CONTINUUM_CACHE_TTL_SECONDS`). Self-invalidating by construction: changing a prompt or schema changes the hash, so stale results can't survive a prompt edit. Only successful results are cached. Required a per-test fake Redis fixture (`tests/conftest.py`) once added, since the real thing would otherwise leak cached results across tests reusing the same generic mock inputs.
- **Live-verified against the real frontend dev server**, not just pytest: `frontend/.env.local` pointed at this backend with `VITE_USE_MOCK_DATA=false`, CORS preflight and an actual cross-origin fetch confirmed from `localhost:5173`, and every response checked field-by-field against the frontend's own TypeScript interfaces rather than just the markdown contract.
